# 2.5D Migration Spec — Full 3D Physics (avian3d)

## Concept

Full switch to avian3d. Physics operates in 3D (XZ floor plane, Y locked). All colliders become 3D volumes (cylinders, cuboids) — NOT spheres. Rendering uses `bevy_sprite3d` for billboard sprites on a perspective camera.

```
        Camera (perspective, from .ron config)
          \
           \
            v
    ┌────────────────────┐
    │   floor (XZ plane) │  ← physics + game logic
    │                    │
    │   ╔═╗  ╔═╗        │  ← billboard sprites (face camera)
    │   ║A║  ║B║        │     colliders are cylinders around them
    │   ╚═╝  ╚═╝        │
    └────────────────────┘
```

## Why Full 3D Over Hybrid 2.5D

- No coord sync layer — avian3d writes Transform directly in 3D
- PhysicsDebugPlugin works out of the box with Camera3d
- SpatialQuery, raycasts, gizmos — all native 3D
- No "two coordinate systems" bugs
- Cylinders prevent the "fly over/under" problem (uniform height, Y locked)

## Coordinate Mapping

```
Old 2D (avian2d)    →    New 3D (avian3d)
──────────────────────────────────────────
X (horizontal)      →    X (horizontal)       same
Y (vertical/screen) →    Z (depth on floor)   Y→Z rename
(none)              →    Y (height)            LOCKED to 0 for physics
```

No negation. Camera is set up so +Z = "up on screen" (camera looks from -Z toward +Z, tilted down).

### Helper Constants & Conversion Functions

```rust
/// Standard locked axes for all game entities:
/// Y translation locked (stay on floor), X/Z rotation locked (stay upright)
const GAME_LOCKED_AXES: LockedAxes = LockedAxes::new()
    .lock_translation_y()
    .lock_rotation_x()
    .lock_rotation_z();

/// Collider half-height for all cylinders/cuboids.
/// Tall enough that nothing passes over/under, thin enough to not matter visually.
const COLLIDER_HALF_HEIGHT: f32 = 50.0;

/// Projectile floor height — projectiles "fly" above the ground.
const PROJECTILE_FLOOR_Y: f32 = 30.0;

/// Convert game-logic Vec2 (XY plane) to 3D world (XZ plane, Y=0).
fn vec2_to_3d(v: Vec2) -> Vec3 {
    Vec3::new(v.x, 0.0, v.y)
}

/// Extract game-logic Vec2 from 3D world position (XZ → XY).
fn vec3_to_2d(v: Vec3) -> Vec2 {
    Vec2::new(v.x, v.z)
}
```

**Replacement patterns:**

| Before (2D) | After (3D) |
|---|---|
| `position.extend(0.0)` | `vec2_to_3d(position)` |
| `transform.translation.truncate()` | `vec3_to_2d(transform.translation)` |
| `Transform::from_translation(pos.extend(0.0))` | `Transform::from_translation(vec2_to_3d(pos))` |

## Physics Migration (avian2d → avian3d)

### Cargo.toml


### Type Changes

| avian2d | avian3d | Notes |
|---------|---------|-------|
| `LinearVelocity(Vec2)` | `LinearVelocity(Vec3)` | Y component always 0 |
| `Collider::circle(r)` | `Collider::cylinder(r, COLLIDER_HALF_HEIGHT)` | Vertical cylinder |
| `Collider::rectangle(w, h)` | `Collider::cuboid(w, COLLIDER_HALF_HEIGHT * 2.0, h)` | Args: x, y, z |
| `Collider::polyline(v, i)` | 4× `Collider::cuboid()` | Arena wall → 4 cuboids |
| `LockedAxes::ROTATION_LOCKED` | `GAME_LOCKED_AXES` (see above) | Lock Y-translation + X/Z rotation |
| `SpatialQuery::shape_intersections(&shape, Vec2, f32, &filter)` | `SpatialQuery::shape_intersections(&shape, Vec3, Quat, &filter)` | Position is Vec3 (y=0), rotation is Quat::IDENTITY |
| `CollisionStart { collider1, collider2 }` | Same | Unchanged |
| `PhysicsLayer` derive | Same | Unchanged |
| `CollisionLayers` | Same | Unchanged |
| `Sensor`, `CollisionEventsEnabled` | Same | Unchanged |
| `RigidBody::Dynamic/Kinematic/Static` | Same | Unchanged |

### Velocity Conversion Pattern

Every place that sets `LinearVelocity` changes from Vec2 to Vec3:

```rust
// Before (2D):
velocity.0 = direction.normalize() * speed;           // Vec2
velocity.0 = Vec2::ZERO;

// After (3D):
velocity.0 = direction.normalize() * speed;           // Vec3 (y=0)
velocity.0 = Vec3::ZERO;
```

The `direction` variables themselves change from Vec2 to Vec3. Computed as:
```rust
// Before:
let direction = (target_pos - my_pos).normalize();     // Vec2

// After:
let direction = (target_pos - my_pos).normalize();     // Vec3, but y=0 naturally
// because both positions have y=0 (locked)
```

### Position Access Pattern

```rust
// Before (2D) — extract Vec2 from Transform:
let pos = transform.translation.truncate();   // → Vec2(x, y)

// After (3D) — use Vec3 directly or extract XZ:
let pos = transform.translation;              // → Vec3(x, 0, z)
let pos_2d = vec3_to_2d(transform.translation);  // → Vec2(x, z) for game logic
```

### Spawn Position Pattern

```rust
// Before (2D):
Transform::from_translation(position.extend(0.0))  // Vec2 → Vec3(x, y, 0)

// After (3D):
Transform::from_translation(vec2_to_3d(position))  // Vec2 → Vec3(x, 0, y)
```

**Decision**: `SpawnSource` target positions stay Vec2 internally. Convert at spawn/physics boundary via `vec2_to_3d()`. This means zero changes in the expression system.

### Cursor → World Raycast

Current `player_input.rs` uses `camera.viewport_to_world_2d()` which doesn't exist for Camera3d. Must raycast from camera through screen point onto the XZ floor plane (Y=0):

```rust
fn cursor_to_world(
    camera: &Camera,
    camera_transform: &GlobalTransform,
    cursor_position: Vec2,
) -> Option<Vec2> {
    let ray = camera.viewport_to_world(camera_transform, cursor_position).ok()?;
    // Intersect ray with Y=0 plane
    if ray.direction.y.abs() < f32::EPSILON { return None; }
    let t = -ray.origin.y / ray.direction.y;
    let world_pos = ray.origin + ray.direction * t;
    Some(Vec2::new(world_pos.x, world_pos.z))  // back to game-logic Vec2
}
```

Returns Vec2 in game-logic coordinates (XZ mapped to XY) — drop-in replacement for the old `viewport_to_world_2d` call. Used in `player_input.rs` for aim direction and targeting.

### Arena Wall

4 static cuboid colliders forming the arena rectangle (sharp corners):

```rust
fn spawn_arena_walls(mut commands: Commands) {
    let hw = ARENA_HALF_WIDTH;
    let hh = ARENA_HALF_HEIGHT;
    let thickness = 20.0;
    let wall_height = COLLIDER_HALF_HEIGHT * 2.0;

    let walls = [
        (Vec3::new(0.0, 0.0, hh), Vec3::new(hw * 2.0, wall_height, thickness)),     // top
        (Vec3::new(0.0, 0.0, -hh), Vec3::new(hw * 2.0, wall_height, thickness)),    // bottom
        (Vec3::new(-hw, 0.0, 0.0), Vec3::new(thickness, wall_height, hh * 2.0)),    // left
        (Vec3::new(hw, 0.0, 0.0), Vec3::new(thickness, wall_height, hh * 2.0)),     // right
    ];

    for (pos, size) in walls {
        commands.spawn((
            Transform::from_translation(pos),
            RigidBody::Static,
            Collider::cuboid(size.x, size.y, size.z),
            CollisionLayers::new(GameLayer::Wall, LayerMask::ALL),
        ));
    }
}
```

## Camera

### Configuration (.ron)

Camera parameters loaded from a `.ron` config for easy tuning without recompile:

```ron
// assets/camera.ron
(
    angle_deg: 55.0,
    distance: 1200.0,
)
```

```rust
#[derive(Resource, Asset, Deserialize, Clone)]
struct CameraConfig {
    angle_deg: f32,     // tilt from horizontal (55-60)
    distance: f32,      // distance from focus point
}

fn camera_offset(config: &CameraConfig) -> Vec3 {
    let angle = config.angle_deg.to_radians();
    Vec3::new(
        0.0,
        config.distance * angle.sin(),      // height
        -config.distance * angle.cos(),     // behind the scene
    )
}
```

### Setup

```rust
fn setup_camera(mut commands: Commands, config: Res<CameraConfig>) {
    let offset = camera_offset(&config);

    commands.spawn((
        Camera3d::default(),
        Transform::from_translation(offset)
            .looking_at(Vec3::ZERO, Vec3::Y),
    ));
}
```

### Camera Follow

```rust
fn camera_follow(
    player: Query<&Transform, With<Player>>,
    mut camera: Query<&mut Transform, (With<Camera3d>, Without<Player>)>,
    config: Res<CameraConfig>,
) {
    let Ok(player_tf) = player.single() else { return };
    let Ok(mut cam_tf) = camera.single_mut() else { return };
    let offset = camera_offset(&config);

    let target = player_tf.translation + offset;
    cam_tf.translation = target;
    // Rotation stays fixed (set once at spawn via looking_at)
}
```

## Sprite Rendering (bevy_sprite3d)

Use `bevy_sprite3d` crate for rendering 2D sprites in 3D space. Handles quad mesh generation and built-in billboarding.

### Setup

```rust
app.add_plugins(Sprite3dPlugin);
```

### Sprite Creation

```rust
// Before (2D):
Sprite { image: texture, custom_size: Some(size), .. }

// After (3D via bevy_sprite3d):
Sprite3dBuilder::from_image(texture)
    .with_pixels_per_metre(1.0)
    .with_billboard(true)         // auto-faces camera
    .with_unlit(true)             // flat shading like 2D
    .build(&mut commands, &mut meshes, &mut materials)
```

### Type Mapping

| Current (2D) | New (3D) | Notes |
|---|---|---|
| `Sprite { image, custom_size }` | `Sprite3dBuilder::from_image(image).with_billboard(true)` | bevy_sprite3d handles mesh+material |
| `Mesh2d(Circle)` + `MeshMaterial2d(ColorMaterial)` | `Mesh3d(Circle)` + `MeshMaterial3d(StandardMaterial { unlit: true })` | Procedural shapes stay manual |
| `Mesh2d(RegularPolygon)` + `MeshMaterial2d(ColorMaterial)` | `Mesh3d(RegularPolygon)` + `MeshMaterial3d(StandardMaterial { unlit: true })` | Triangle shape |

All non-sprite materials use `unlit: true` to match current flat 2D look.

Fallback if `bevy_sprite3d` doesn't support Bevy 0.18: manual `Mesh3d(quad)` + `MeshMaterial3d(StandardMaterial { unlit: true })` + custom Billboard component (see Risks).

### Sprite Pivot — Bottom-on-Floor

All entities use bottom-on-floor pivot. Sprite child's local Y offset = half sprite height, so the bottom edge touches Y=0 (the floor).

```rust
// Character/mob sprites:
Transform::from_translation(Vec3::new(0.0, sprite_height / 2.0, 0.0))
// sprite_height = custom_size.y from the Sprite blueprint component
```

Projectiles also use bottom-on-floor, but their root entity sits at an elevated Y:

```rust
// Projectile root entities: Y = PROJECTILE_FLOOR_Y instead of Y = 0
Transform::from_translation(Vec3::new(x, PROJECTILE_FLOOR_Y, z))
```

Their `LockedAxes` still locks Y-translation (at the elevated value). Sprite pivot logic is the same (bottom-on-floor relative to their elevated Y).

Summary:
- Characters/mobs: root at Y=0, sprite bottom at Y=0
- Projectiles: root at Y=PROJECTILE_FLOOR_Y, sprite bottom at PROJECTILE_FLOOR_Y
- Shadows: always at world Y=0.01 (on the actual floor) regardless of entity's root Y

## Entity Structure

### Before (2D)
```
Root (Transform XY, RigidBody::Dynamic, Collider::circle)
  └─ Visual child
      ├─ Sprite (Bevy 2D Sprite at Z=0)
      └─ Shadow (Mesh2d ellipse at Z=-0.001)
```

### After (3D)
```
Root (Transform XZ, Y=0 or PROJECTILE_FLOOR_Y, RigidBody, Collider::cylinder, GAME_LOCKED_AXES)
  └─ Visual child
      ├─ Sprite (bevy_sprite3d billboard, local Y = height/2)
      └─ Shadow (Mesh3d ellipse flat on floor, local Y = -root.Y + 0.01)
```

Shadow local Y compensates for the parent's floor height so the shadow always renders on the actual floor (Y~0).

## Animation Systems

### System Ordering

bevy_sprite3d billboard rotation runs in `PostUpdate`. All game animation systems run in `Update` and modify local-space child transforms. Local offsets and scales are applied before billboard world rotation — animations compose correctly.

### SquishWalkAnimation
Modifies child `transform.scale` and `transform.translation.y`. Since the child sprite is a billboard quad oriented vertically, scaling X/Y of the child still stretches width/height. **Works as-is.**

### JumpWalkAnimation
Adds local Y displacement (bounce) — still Y in 3D (lifts sprite off floor). Local Z rotation (tilt) — on a billboard, the tilt axis is the billboard's local Z (pointing at camera). **Works as-is** — local transforms in Update apply before billboard world rotation in PostUpdate.

### HitFlash
Scale + color manipulation. **No changes needed.**

## Falling Component

`falling.rs` is unique: it uses Y as true vertical height (parabolic arc to simulate a projectile falling from above). Needs explicit coordinate mapping:

```rust
// Before (2D) — Y is both screen-vertical and game-vertical:
transform.translation.x = falling.target_position.x;
transform.translation.y = start_y - (falling.height * eased_t);

// After (3D) — horizontal on XZ, vertical on Y:
transform.translation.x = falling.target_position.x;      // game X → world X
transform.translation.z = falling.target_position.y;       // game Y → world Z
transform.translation.y = start_y - (falling.height * eased_t);  // vertical arc on world Y
```

Falling entities must NOT lock Y-translation (unlike all other entities) since they need vertical movement for the arc. Use separate locked axes without `lock_translation_y()`. After landing (arc complete), the entity is despawned so re-locking is not needed.

`TargetInfo::from_position(falling.target_position)` stays Vec2 — no change needed (expression boundary).

## Orbiting Component

`orbiting.rs` computes circular positions via `sin`/`cos` and `.extend(0.0)`. Needs XZ mapping:

```rust
// Before (2D):
let offset = Vec2::new(angle.cos(), angle.sin()) * radius;
transform.translation = owner_pos + offset.extend(0.0);

// After (3D):
let offset = Vec3::new(angle.cos() * radius, 0.0, angle.sin() * radius);
transform.translation = owner_pos + offset;
// owner_pos is already Vec3 from Transform, Y=0
```

Both `init_orbiting` and `update_orbiting_positions` need this change.

## Damage Numbers

### 3D Position Mapping

Spawn position changes from `Vec3(x, y, z_layer)` to true 3D:

```rust
// Before (2D):
let position = event.position + Vec3::new(0.0, 20.0, Z_LAYER);
// event.position is Vec3 from Transform, Z_LAYER=100 for draw order

// After (3D):
let position = Vec3::new(event.position.x, TEXT_FLOAT_HEIGHT, event.position.z);
// TEXT_FLOAT_HEIGHT = enough above sprites (~60-80)
// event.position is Vec3 from Transform (x, 0, z)
```

### Parabolic Motion in 3D

`ParabolicLens` currently animates `translation.x` (horizontal drift) and `translation.y` (vertical arc). In 3D, Y stays as vertical (correct), but horizontal drift should spread on XZ:

```rust
struct ParabolicLens {
    start: Vec3,
    drift_x: f32,       // was vx — horizontal drift on world X
    drift_z: f32,       // NEW — horizontal drift on world Z
    vy: f32,            // vertical velocity (world Y) — unchanged
}

impl Lens<Transform> for ParabolicLens {
    fn lerp(&mut self, mut target: Mut<Transform>, ratio: f32) {
        let t = ratio * DURATION;
        target.translation.x = self.start.x + self.drift_x * t;
        target.translation.z = self.start.z + self.drift_z * t;
        target.translation.y = self.start.y + self.vy * t - 0.5 * GRAVITY * t * t;
    }
}
```

`drift_x` and `drift_z` are random spread in both horizontal axes (instead of only X).

## Expression System Impact

`VecExpr` currently evaluates to `Vec2`. This is used for targeting directions, spawn positions, etc.

**Decision: keep VecExpr as Vec2 internally.** Convert at the spawn/physics boundary:
```rust
// In spawn.rs or wherever VecExpr result is used:
let dir_2d: Vec2 = vec_expr.eval(source, stats);
let dir_3d = vec2_to_3d(dir_2d);
```

This means the entire expression system (`src/expr/`), all `.ron` files, all targeting logic — **zero changes**. Only the boundary where Vec2 enters physics/transform needs conversion.

Similarly for spawn positions in SpawnSource — keep as Vec2 in the source, convert to Vec3 at the point of spawning.

## Risks & Mitigations

| Risk | Mitigation |
|------|-----------|
| Cylinder colliders behave differently than circles | Height is locked, Y is locked — effectively 2D. Test edge cases. |
| Camera angle makes sprites unreadable | CameraConfig in .ron — tweak without recompile. Start with ~55°. |
| bevy_sprite3d incompatible with Bevy 0.18 | Fallback: manual `Mesh3d(quad)` + `MeshMaterial3d(StandardMaterial { unlit })` + custom Billboard component/system. |
| Text2d doesn't render with Camera3d | Fallback: bevy_sprite3d billboard with text texture, or screen-space UI overlay. |
| Performance: 3D physics heavier than 2D | Minimal — same entity count, cylinders are simple primitives. |
| Falling entities need unlocked Y | Use separate LockedAxes without lock_translation_y(). Entity despawns after arc completes. |

## Decisions (Resolved)

1. **Camera angle** → Loaded from `assets/camera.ron` (start with 55°, tunable without recompile)
2. **Arena wall shape** → 4 cuboids (sharp corners), no trimesh
3. **Sprite pivot** → Bottom-on-floor for ALL entities. Projectiles use elevated floor (Y = PROJECTILE_FLOOR_Y ≈ 30)
4. **Sprite rendering** → `bevy_sprite3d` crate for sprites + billboarding. Fallback to manual Mesh3d if incompatible with Bevy 0.18
